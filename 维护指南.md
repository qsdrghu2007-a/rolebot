# Telegram 聊天机器人 — 项目维护指南

## 架构概览

```
main.py                        # 入口：配置加载、依赖检查、启动 bot
telegram_bot.py                # 核心：命令处理、消息流、心情系统、callback 分发
llm_client.py                  # LLM API 客户端（OpenAI 兼容，支持任意 base_url）
prompt_engine.py               # 提示词引擎（KV 缓存优化、mood 注入）
memory_db.py                   # SQLite 数据库（6 表 + mood/erotic/compact CRUD）
ai_memory_manager.py           # AI 记忆判断 + 性欲 ArousalJudge
world_book.py                  # 世界书关键词匹配
setup.py                       # 首次配置引导（交互式，支持任意 LLM 服务商）
config.yaml                    # 运行时配置（llm 部分：api_key / base_url / model）
config.example.yaml            # 配置模板
digital_person/
  persona_final-d.md           # 角色人格设定（模板，需用户自行填写）
  world_book.json              # 世界书示例条目
```

> **注意**：此项目为通用模板，不包含任何特定角色内容。用户需通过 `persona_final-d.md` 自定义角色，通过 `config.yaml` 配置 LLM 连接。

## 数据流

```
用户消息
  → handle_text_message()
    → 检测 reply_to_message? → _handle_edit（编辑模式）
    → _get_mood_state()       → 懒加载时间衰减
    → build_messages(mood)    → messages[0]=persona, [1..N-1]=历史, [N]=WB+心情+用户消息
    → generate_response()     → 同步 LLM 调用
    → _release_missing_if_expressed() / _track_response()
    → add_message(user + assistant) 到 DB
    → create_task → _process_memory_async（记忆+心情增减）
    → create_task → _process_arousal_async（文爱状态机）
```

## KV 缓存结构

```
messages[0]      persona + 长期记忆    ← 固定 → 缓存命中
messages[1..N-1] 对话历史              ← 旧消息不变 → 缓存命中
messages[N]      WB + 心情 + 焦点提醒 + 用户消息 ← 每轮变化
```

## 心情系统（5 维 0-10）

### 懒加载时间衰减（收到消息时计算距上次更新的时间差）
| 维度 | 默认 | 衰减公式 | 说明 |
|------|------|---------|------|
| happiness | 7 | -0.1/h | 不聊缓慢下降 |
| missing | 3 | +0.15/h | 越不聊越想（唯一随时间涨的维度）|
| energy | 7 | 深夜-1 / 早晨+0.5 / 其他-0.1/h | 模拟作息 |
| anger | 1 | -0.3/h | 自己消气 |
| arousal | 3 | +2.0/h | 5h 从 0 涨满 |

### 减少 missing 的两个通道
1. **AI 判断**：用户回来/聊得开心 → missing -1 到 -2
2. **表达释放**：bot 回复含"想你""想念""好想你""想你了"等 → missing -0.5

### 注入位置
`prompt_engine.py: _build_user_content()` → messages[N] 末尾
格式：`【你的内在状态】你现在{心情描述}（不要直接说出来，用来影响语气）`

> **通用版差异**：心情描述使用中性表述（如"非常想念对方""很想要亲密互动"等），不使用特定角色称呼。

## 文爱状态机

### ArousalJudge（独立 API）
- 模型：deepseek-v4-flash, temp=0, max_tokens=10, thinking=disabled
- 只判 true/false：**明确性行为/dirty talk/文爱场景** → true
- 亲亲抱抱安抚暧昧 → false

### 状态机
```
进入：连续 2 次 true → active=true, arousal=10
退出：连续 3 次 false → active=false, arousal=0
重置：active 模式下任何 true → count_exit=0
```

### 各命令的文爱判断触发规则

| 路径 | 触发 arousal？ | 理由 |
|------|:---:|------|
| 正常聊天 `_process_message` | ✓ | 新用户消息，必须判断 |
| `/continue` | ✗ | 无新用户消息，bot 自己续写 |
| `/regenerate` | ✗ | bot 重生成自己的回复，用户消息未变 |
| `/suggest` 选择 | ✓ | 选中文本作为用户发言存入 DB，需判断 |
| `_handle_edit` 编辑自己消息 | ✓ | 编辑后内容是新用户消息，需判断 |
| `_handle_edit` 编辑 bot 消息 | ✗ | 编辑目标非用户消息，不触发 |

### 文爱状态快照系统

每条用户消息判定后，状态机的当前状态（`active`、`enter_count`、`exit_count`）写入 `conversations` 表的 `erotic_*` 三列。bot 消息行不写入。

**快照的作用**：编辑消息时恢复状态机。

**编辑流程（`_handle_edit`）**：

```
1. 如果是编辑用户自己的消息 → reset_arousal_snapshot(db_id)  // 旧判定作废
2. edit_message_content(db_id, reply_text)                      // 改写内容
3. delete_messages_from_id(db_id + 1)                           // 后面的消息+快照一起删
4. prev = get_last_arousal_snapshot_before(user_id, db_id)      // 查编辑点之前的最近快照
5. save_erotic_state(prev)                                       // 状态机回滚到编辑点之前
6. 如果触发重生 且 编辑的是用户消息：
   → _process_arousal_async(new_content, user_msg_db_id)        // 判断新内容+写新快照
```

**快照查询方法**（`memory_db.py`）：

| 方法 | 作用 |
|------|------|
| `set_arousal_snapshot(conv_id, active, enter, exit)` | 写三列 |
| `get_last_arousal_snapshot_before(user_id, before_id)` | 查指定 ID 之前最近一条 user 消息的快照 |
| `reset_arousal_snapshot(conv_id)` | 置 NULL（编辑后旧判定失效） |

**状态机的双计数器语义**：

| erotic_active | enter_count | exit_count |
|:---:|:---:|:---:|
| 0（未进入） | 活跃计数器 | 固定为 0 |
| 1（已进入） | 固定为 0 | 活跃计数器 |

永远不同时改两个 count，由 `active` 决定改哪个。转换时两计数器归零。

## 设计决策

### 为什么用 SQLite 而不是 MySQL/PostgreSQL
- 零配置、零依赖服务
- 单用户/小规模场景足够
- 数据库文件可直接备份

### 为什么 LLM 调用是同步的
- 使用 `asyncio.get_event_loop().run_in_executor()` 包装同步 requests 调用
- 兼容性好，不需要额外异步 HTTP 库
- 主回复完成前用户看不到 typing 结束，流式输出未见明显体验提升

### 为什么思考模式默认开启
- DeepSeek 等模型的 thinking 模式回复质量明显更高
- 用户可通过 `/thinking off` 关闭以换取速度

## 已知坑

1. **`delete_messages_from_id` 曾丢失** — 编辑功能依赖此方法，需确保 memory_db.py 中存在（第 657 行附近）。
2. **时间戳兜底** — `clear_command` 和 `_do_compact` 都用 `get_recent_conversation` 做兜底。
3. **心情注入改过** — 从 `【当前心情】` 改为 `【你的内在状态】` 指令形式。
4. **表情符号数据库** — `world_book.py` 同时兼容 `keys` 和 `keywords` 字段，JSON 可以是数组或 `{"entries": [...]}` 对象。
5. **首次运行** — `python setup.py` 引导配置。config.yaml 和 persona_final-d.md 都是模板，开机前必须填好。
6. **文爱快照只在 user 行有值** — `set_arousal_snapshot` 仅对 `role='user'` 的 conversation 行调用。bot 行的三列为 NULL。
7. **编辑 bot 消息也恢复状态机** — 即使编辑目标不是用户消息，`_handle_edit` 仍会恢复到编辑点之前的快照。因为删除后续消息时可能删掉了中间的 user 快照。
